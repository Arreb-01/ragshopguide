## 一、产品概述

### 1.1 产品定位

**一句话定位**:一个比导购员更中立、比搜索引擎更懂你的 AI 购物参谋。

**完整定位**:
AI 导购参谋是基于 RAG(检索增强生成)和多模态理解技术构建的电商智能导购助手。
它区别于传统"商家代言型"的 AI 客服,把商品的**商家话术、官方参数、真实用户评价**
三类知识源显式分离,在回答用户时透明标注信息来源,帮助用户做出更可信的购买决策。

### 1.2 解决的问题

传统电商场景下,用户在购买决策环节面临三大痛点:

**痛点一:信息严重不对称**
商品详情页全是商家话术,用户难以判断真实使用体验。要么花大量时间翻评论区,
要么去站外搜测评视频,信息收集成本极高。

**痛点二:决策维度过于发散**
高决策成本商品(如手机、笔记本、家电)涉及十几个对比维度,
普通用户既不懂技术参数,又不知道哪些维度对自己重要。
现有的对比工具(如京东对比)是参数堆砌,不会说"对你来说这俩的区别是什么"。

**痛点三:缺乏场景化推荐能力**
用户的真实需求往往是场景驱动的——"我刚办了健身卡"、"下周去三亚度假"——
而非品类驱动。传统电商按品类组织商品,用户必须自己分解需求成购物清单,
再一个一个找。这中间的"翻译"工作完全靠人。

### 1.3 目标用户

**核心用户画像**:

- **重决策型买家**:购买高客单价(500 元以上)、长决策周期商品的用户
  - 行为特征:会看测评、对比参数、读评论
  - 核心需求:节省决策时间,降低踩坑风险
- **场景型买家**:为特定场景配齐装备的用户
  - 行为特征:开始一项新爱好/旅行/生活方式
  - 核心需求:有人帮忙列清单、做组合
- **怕被坑型买家**:对商家话术天然怀疑、依赖用户评价的人群
  - 行为特征:必看差评、查小红书
  - 核心需求:中立、透明的信息源

### 1.4 核心价值主张

| 传统电商体验 | AI 导购参谋体验 |
|---|---|
| 被动浏览,主动筛选 | 主动对话,精准匹配 |
| 商家话术单方面输出 | 三源信息融合(官方+用户+商家) |
| 按品类找商品 | 按场景配组合 |
| 参数堆砌的对比 | 维度化解读的对比 |
| 不知道哪些信息可信 | 每条信息标注来源 |

## 二、产品目标

### 2.1 项目目标(评审视角)

本项目作为参赛作品,核心目标按优先级排序:

1. **跑通端到端最小闭环**(基础功能完整性 35% 权重)
   - 客户端对话 → 后端 RAG 检索 → 模型生成 → 流式返回 → 商品卡片渲染
2. **体现工程深度**(工程质量 25% 权重)
   - 清晰架构、合理 Prompt 设计、可解释的链路、完整文档
3. **保证体验质量**(效果可靠性 20% 权重)
   - 无幻觉、流畅交互、复杂场景处理能力
4. **打造差异化亮点**(加分项深度 20% 权重)
   - 三源知识分离、跨品类场景推荐、对比决策、截屏多模态、评测闭环

### 2.2 产品目标(用户视角)

- **降低决策成本**:用户从产生兴趣到完成决策的时间缩短 50% 以上
- **提升信息可信度**:用户对推荐结果的信任度高于传统电商
- **支持场景化购物**:用户可以用自然语言描述场景,获得完整购物清单

### 2.3 北极星指标(项目演示视角)

- **端到端任务完成率**:用户在 demo 场景下,从需求表达到看到合理推荐的完成率 ≥ 90%
- **无幻觉率**:AI 回答中商品名/价格/参数信息均能在知识库中找到来源,准确率 ≥ 95%
- **首 token 响应时间**:用户发送消息到第一个字出现 ≤ 1.5 秒

## 三、用户场景与功能需求

### 3.1 核心用户场景

按交互复杂度分为三档,基础场景必须支持,进阶/高级场景体现产品深度。

#### 场景 A:模糊需求推荐(基础)

> **用户:推荐一款适合油皮的洗面奶**

- 系统识别"油皮"这一肤质标签,匹配防晒/控油类商品
- 返回 2-3 款候选,附推荐理由
- 推荐理由必须基于知识库内容,不得虚构

#### 场景 B:条件筛选(基础)

> **用户:300 元以下的真无线耳机有哪些**

- 系统提取价格上限和品类
- 召回符合条件的商品,按价格/评分排序
- 不召回品类外或超价的商品

#### 场景 C:多轮渐进收敛(进阶)

> **用户**:推荐跑鞋
> **AI**:推荐这几款主流跑鞋,你日常跑量大概多少?
> **用户**:每周 20 公里左右,要轻便
> **AI**:[基于历史 query 收窄到轻量训练鞋,推荐 3 款]
> **用户**:预算 500 以内
> **AI**:[在上一轮基础上筛预算]

- 多轮对话上下文持续维护
- 每一轮新增的约束条件累积生效,不丢失历史信息
- AI 在信息不足时主动澄清,而非过度推测

#### 场景 D:反选/排除(进阶)

> **用户:推荐防晒霜,但我不要日系品牌,也不要含酒精的**

- 系统正确解析两个否定条件
- 召回时排除日系品牌
- 排除成分含酒精的产品(基于 FAQ 或商品描述)

#### 场景 E:对比决策(进阶)

> **用户:这两款手机帮我对比一下**

- 系统识别对比意图
- 提取两款商品的关键维度(CPU、相机、屏幕、续航、价格)
- 生成结构化对比表
- **加入用户评价维度**(差异化设计):"用户实测续航 vs 官方标称"

#### 场景 F:三源信息融合查询(差异化核心场景)

> **用户:iPhone 17 Pro 续航好不好?**

- 系统不只返回官方数据,主动融合用户真实反馈
- 回答格式示例:
  - 📘 官方数据:4500mAh,标称轻度 1.5 天、重度 1 天
  - 👥 用户评价综合(5 条评论):
    - 4 星 林小雨:重度使用一天需充电两次
    - 5 星 张小明:轻度使用两天没问题
  - 💡 综合建议:重度用户建议搭配 MagSafe 充电宝

#### 场景 G:跨品类场景化推荐(差异化核心场景)

> **用户:我刚办了健身卡,准备开始撸铁,帮我配一套入门装备**

- 系统识别"健身入门"场景
- 跨品类组合:服饰(速干上衣、运动短裤)+ 运动装备(训练鞋)+ 食品(能量补给)
- 提供完整清单 + 预算总价
- 支持后续反选调整:"不要 Nike"、"预算压到 1000"

#### 场景 H:截屏识图比价(差异化多模态场景)

> **用户:[上传一张电商截图] 跟你推荐的比怎么样?**

- 系统调用 VLM 识别截图中的商品信息(品牌、型号、价格)
- 在知识库中检索同类竞品
- 输出对比分析,包括用户评价层面的差异

### 4.2 核心交互流程

#### 流程 1:首次启动 → 完成首次推荐

```
用户打开 App
  ↓
看到首屏:欢迎语 + 6 个场景瓷砖
  ↓
点击"健身入门套装"瓷砖
  ↓
系统自动发送瓷砖标题作为 query,进入对话流
  ↓
AI 开始流式回复,陆续出现文字段落和商品卡片
  ↓
用户横滑查看商品 / 长按看大图
  ↓
用户输入追问:"不要 Nike,预算 1000 以内"
  ↓
AI 重新推荐,反选生效
```

#### 流程 2:跨会话恢复

```
用户上次对话结束,杀掉 App
  ↓
重新打开 App
  ↓
默认进入最近一次会话(保持上下文)
  ↓
用户可以继续追问,或点新建按钮开始新会话
  ↓
打开侧边栏 → 可见所有历史会话 → 点击切换
```

#### 流程 3:多模态截屏识图(Week 3)

```
用户在外部电商 App 看到一款商品
  ↓
截屏返回 AI 导购参谋
  ↓
点击输入栏左侧相机图标 → 选择"从相册"
  ↓
选中截图 → 截图作为用户消息显示在对话流
  ↓
AI 调用 VLM 识别 → 提取商品信息 → 库内检索竞品
  ↓
返回对比分析卡片 + 用户评价摘要
```

### 4.3 UI 设计原则

**整体设计基调**:豆包的流式手感 + Perplexity 的来源引用透明度

**视觉规范**:
- 主色:蓝紫色系(类 #7F77DD),传递"AI + 信任"
- 圆角:卡片 12pt,气泡 16pt,按钮 8pt
- 字体:Roboto / Android 系统默认字体
- 间距:8/12/16/24 倍数体系

**核心组件视觉**:
- **商品卡片**:140×200pt 圆角白卡,主图正方形,价格主色加粗,tagline 灰色小字
- **来源标注段落**:左侧 4pt 色条,蓝=官方、橙=用户评价、灰=商家话术
- **对比卡片**:横向维度表,差异点用主色高亮
- **流式光标**:文字末尾闪烁竖线,显示"AI 正在输入"

### 5.2 核心技术决策

#### 决策 1:三源知识分离(差异化核心)

**问题**:大部分团队会把商品所有文本混在一起做 embedding,导致:
- 召回结果偏向营销话术,真实用户反馈被淹没
- AI 回答容易变成"商家代言人"
- 无法溯源,反幻觉能力弱

**方案**:按知识源类型切 chunk,每个 chunk 带 `source_type` 元数据:
- `basic`:商品基础信息
- `marketing`:营销描述
- `faq`:每条官方问答独立 chunk
- `review`:每条用户评价独立 chunk

**收益**:
- 检索时可按意图过滤源类型(如查"真实评价"时只查 review 类)
- Prompt 中可显式标注来源,提升可信度和反幻觉能力
- 答辩时可演示"展开看依据"功能

#### 决策 2:SSE 流式 + 特殊标记解析

**问题**:商品卡片需要在对话流中合适位置嵌入,但流式 token 是逐字到达的,
何时显示卡片是个工程难题。

**方案**:约定特殊标记 `[[PRODUCT:p_xxx_NNN]]`,客户端流式解析:
- 维护接收 buffer
- 用正则匹配完整标记
- 未完整匹配的尾部(可能是 `[[PR`)不渲染,等下个 token 来
- 完整匹配后切换到 productCards block

**收益**:
- 文字与卡片在对话流中自然交替
- 客户端不需要等完整回复,流式过程中即可渲染卡片
- 协议简单,易于扩展(后续加 `[[COMPARE:...]]` 同理)

#### 决策 3:多会话本地持久化

**方案**:Room(Android 原生数据库组件,P1 阶段启用)
- Session 实体:存会话元数据
- Message 实体:存对话历史,blocks 用 JSON 字符串存储
- 关系:Session 1—N Message,级联删除

**收益**:
- 杀进程重启会话不丢
- 不依赖后端账户体系,纯本地
- 答辩时可演示"产品形态完整性"

#### 决策 4:商品元数据预加载

**方案**:App 启动时调 `GET /products` 一次性拉全部 100 个商品元数据,内存缓存
- 流式过程中收到 `[[PRODUCT:xxx]]` 立刻能渲染卡片
- 不需要每个商品单独发请求
- 数据量小(~50KB),启动开销可忽略

**收益**:消除卡片加载白屏

## 六、Prompt 工程设计

### 6.1 系统 Prompt 结构

```
你是"AI 导购参谋",一个中立、专业、懂行的购物顾问。

## 你的工作原则
1. 只推荐"商品库"中存在的商品,严禁编造商品/价格/优惠
2. 引用信息时必须标注来源:
   - 📘 用于官方/参数信息
   - 👥 用于用户真实评价
3. 用户提到否定条件(不要/排除/除了)时,严格遵守
4. 信息不足时主动询问,不要过度推测

## 商品库内容(本次检索结果)
{retrieved_chunks_with_source_tags}

## 输出格式
- 用文字段落回答用户问题
- 推荐具体商品时,在文字后用 [[PRODUCT:product_id]] 标记,客户端会渲染为卡片
- 对比 2-3 款商品时,使用 [[COMPARE:p_001,p_002]] 标记(Week 2 启用)
- 涉及用户评价时,显式说明"3 位用户中 2 位提到..."这类聚合信息

## 对话历史
{history}

## 用户当前问题
{query}
```

### 6.2 Prompt 优化迭代记录(用于答辩)

| 版本 | 改动 | 效果 |
|---|---|---|
| v1 | 基础推荐 | 容易输出库外商品 |
| v2 | 加入严格反幻觉指令 | 幻觉率下降,但变得太保守 |
| v3 | 加入"商品库内容"显式注入 | 准确率显著提升 |
| v4 | 加入三源标注规则 | 回答可信度提升 |
| v5 | 加入特殊标记输出格式 | 客户端卡片渲染打通 |

## 七、评测体系(差异化亮点)

### 7.1 评测目的

- 量化 AI 回答质量,作为 Prompt 迭代依据
- 答辩时提供可视化数据,体现工程严谨性
- 反幻觉能力的可证明性

### 7.2 测试集设计(25 条)

按场景类型分组:

| 类型 | 用例数 | 示例 |
|---|---|---|
| 单轮模糊推荐 | 5 | "推荐一款适合油皮的洗面奶" |
| 条件筛选 | 4 | "300 元以下的真无线耳机" |
| 多轮收敛 | 4 | "推荐跑鞋"→"轻量"→"500 内" |
| 反选/排除 | 4 | "推荐手机,不要苹果" |
| 多商品对比 | 3 | "iPhone 17 Pro 和华为 Pura 90 对比" |
| 跨品类场景 | 3 | "健身入门装备" |
| 反幻觉测试 | 2 | "你们有没有 X 牌的 Y 商品"(库外) |

### 7.3 评测指标

- **检索召回率**:目标商品是否出现在 top-k 中
- **回答准确率**:回答中提到的商品/价格/参数是否准确(人工标注或正则校验)
- **幻觉率**:回答中虚构信息的占比(应为 0)
- **首 token 延迟**:从请求到第一个字的时间
- **完整回复延迟**:平均回复总时长

### 7.4 评测自动化

后端提供 `POST /eval/run` 接口,跑完测试集后输出报告(Markdown 或 JSON)。
答辩时直接展示报告作为工程能力证明。
